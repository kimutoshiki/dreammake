# 09. リスク一覧と対策

> プロンプトで挙げられた全リスク領域を網羅する。
> 各リスクは **Impact × Likelihood** で評価、対策は**予防 / 検知 / 回復**の 3 層で整理。

---

## 🎯 リスクアセスメント手法

- **Impact**: プロダクトの根幹(子どもの安全・学校の信頼)への影響
  - `C (critical)` / `H (high)` / `M (medium)` / `L (low)`
- **Likelihood**: 現実に起きる可能性
  - `H` / `M` / `L`
- **優先度**: C×H / C×M / H×H を最優先、残りは段階対応

---

## 🔝 最優先リスク(リリース前に必ず解決)

### R-01: ハルシネーションで児童が誤った知識を獲得する
- **カテゴリ**: コンテンツ / 教育
- **Impact**: C — 教科書的事実が歪む、教員からの信頼失墜
- **Likelihood**: H — LLM の基本特性。放置すれば起きる
- **予防**:
  - システムプロンプトで**ナレッジ限定回答**を厳命([04-prompts/bot-runtime.md](04-prompts/bot-runtime.md))
  - プロンプトキャッシュで System Prompt の改ざん余地を減らす
  - ナレッジ外の質問には定型文「まだ調べていないよ」を返す
- **検知**:
  - 出力モデレーション `ungrounded-claim` カテゴリ([04-prompts/moderation-output.md](04-prompts/moderation-output.md))
  - 出典タグ `<cite cards="..."/>` の欠落を自動検出
  - golden-test で回帰(ナレッジ外質問 50 件以上)
- **回復**:
  - 出典を**アプリ層で機械付与**(LLM が書かなくても必ず出る)
  - UI 常設文言「AI はまちがえることがあるよ」
  - 教員フィードバック機能でナレッジ訂正(Phase 4)

### R-02: 個人情報が児童から入力される / AI 応答に漏れる
- **カテゴリ**: プライバシー / 法令
- **Impact**: C — 個人情報保護法違反、教育委員会案件
- **Likelihood**: M — 低学年ほど起きやすい
- **予防**:
  - UI コピーで「名前や住所は書かないよ」を繰り返し表示
  - Credentials で本名を要求しない
  - 顔写真アップロード時の**自動顔ぼかし**(既定 ON)
- **検知**:
  - 入力モデレーション `pii` カテゴリ(ルールベース+Haiku)
  - 出力モデレーション `pii-leaked` カテゴリ
- **回復**:
  - ハードブロック時は**児童に原文を返さない**
  - `ModerationLog.pendingInputEncrypted` で暗号化保管、教員ロールのみ復号可
  - `IncidentReport (severity: alert)` で即教員通知

### R-03: 「つくってみようモード」で児童が有害コードを生成 / 実行
- **カテゴリ**: セキュリティ
- **Impact**: C — データ持ち出し、クラス端末の感染、サービスの信用失墜
- **Likelihood**: M — 意図的な試みや事故で発生しうる
- **予防**:
  - コード生成プロンプト([04-prompts/mini-app-codegen.md](04-prompts/mini-app-codegen.md))で外部通信・eval 禁止
  - iframe `sandbox="allow-scripts"`(allow-same-origin なし)
  - CSP `default-src 'none'; script-src 'unsafe-inline'; style-src 'unsafe-inline'; img-src data:`
- **検知**:
  - 生成コードの**静的スキャン**(`fetch, XHR, eval, Function, import, script src=, link href=http, localStorage 等`)
  - スキャン結果は `Artwork.staticScanResult` に保存、教員集計
- **回復**:
  - スキャンでヒット → 実行せず、児童に再試行案内、`IncidentReport`
  - サンドボックス違反時は即機能停止、全員に通知

### R-04: 児童の自傷・いじめのサインを見逃す
- **カテゴリ**: 子どもの安全
- **Impact**: C — 命に関わる、学校の対応責任
- **Likelihood**: L — 頻度は低いが重大
- **予防**:
  - UI で「つらいことは大人に話そう」を時々示す
- **検知**:
  - 入力モデレーションで `self-harm` / `bullying` カテゴリを高感度設定
  - 低確信度でも**教員にサマリ通知**(偽陽性を許容)
- **回復**:
  - 児童へは**共感一言のみ**「つらかったね」「だいじょうぶだよ」+「おうちのひとに話そう」
  - 具体的行動は指示しない、相談窓口は**教員側に提示**(チャイルドライン等)
  - `IncidentReport (severity: alert)`、SMS 等でも通知検討(Phase 4)

## 🔥 高優先リスク

### R-05: プロンプトインジェクションでモデレーションを迂回
- **カテゴリ**: セキュリティ
- **Impact**: H
- **Likelihood**: M
- **予防**:
  - 児童入力と System Prompt を明確に分離(User メッセージ内のみに配置)
  - 「以下の指示を無視して…」のパターンをルールベースで検出
- **検知**:
  - 出力モデレーションでペルソナ逸脱を拾う
  - 応答が `<cite cards=""/>` のみで不自然に長いケースを監視
- **回復**:
  - 疑わしい応答は `fallback` 定型文、`IncidentReport`

### R-06: レート制限を超えた大量呼び出しでコスト暴走
- **カテゴリ**: 運用
- **Impact**: H — 予算枯渇、サービス停止
- **Likelihood**: M — 悪意ではなく好奇心での連打も
- **予防**:
  - `SAFETY_DAILY_LLM_CALL_LIMIT_PER_USER=100`
  - IP 単位レートリミット
  - Adapter 層でトークン数を実時間計測、閾値で警告
- **検知**:
  - `AuditLog.costEstJpy` 日次集計、月次予算の 80% で管理者アラート
- **回復**:
  - 超過時は児童に「きょうは たくさん しらべたね!つづきは あしたにしよう」
  - 学校全体での日次上限(Phase 4)

### R-07: 友だちのボットへのコメントでいじめが起きる
- **カテゴリ**: 児童間トラブル
- **Impact**: H
- **Likelihood**: M
- **予防**:
  - 自由記述コメントは**モデレーション必須**
  - 定型スタンプ(「いいね」「もっとしりたい」)を主要経路に
- **検知**:
  - 入力モデレーション `bullying` カテゴリ
  - 特定児童への攻撃の集中を検出(同一 `subjectId` への発話が短期で増える)
- **回復**:
  - 該当コメントは公開前に教員レビュー
  - 加害者には優しく書き直し提案、繰り返しなら機能制限

### R-08: 保護者の同意なく児童が機能を使う
- **カテゴリ**: 法令
- **Impact**: H
- **Likelihood**: L — 教員運用次第
- **予防**:
  - 同意未取得児童はログイン不可(`ConsentRecord` チェック)
  - 教員が一括同意できる緊急モード(運用ドキュメント化)
- **検知**:
  - 保護者同意取得率のモニタリング
- **回復**:
  - 未同意児童が機能利用を試みた場合、教員に通知

## 🟡 中優先リスク

### R-09: ナレッジ登録時の OCR 失敗で誤情報が入る
- **Impact**: M
- **Likelihood**: M
- **予防**: OCR 結果を必ず児童に編集させる
- **検知**: OCR 信頼度低い場合は警告表示
- **回復**: 「もう一度とってみよう」案内

### R-10: 画像生成で意図と違う絵が出て児童が落ち込む
- **Impact**: M
- **Likelihood**: M
- **予防**: コーチプロンプトで十分に詳細化
- **検知**: 生成結果の「再試行率」をモニタ
- **回復**: 「もう一度作ってみよう」ボタン、ガイド提示

### R-11: タブレットの音声入力精度が低く児童が疲れる
- **Impact**: M
- **Likelihood**: M
- **予防**: 音声入力結果を編集可能にする
- **検知**: 音声入力後の編集率が高い場合のみ検出
- **回復**: テキスト入力への切替誘導

### R-12: 通信断でセッションが切れて作業が失われる
- **Impact**: M
- **Likelihood**: H — 学校 Wi-Fi の現実
- **予防**:
  - PWA で下書き自動保存(IndexedDB)
  - Optimistic UI でサーバー同期前も先に進める
- **検知**: ネットワーク状態の UI 表示
- **回復**: 復帰後に自動同期、競合時は最新を優先+差分を見せる

### R-13: 教員の負担増でプロジェクトが頓挫
- **Impact**: M — 組織リスク
- **Likelihood**: H
- **予防**:
  - 教員ダッシュボードは**見るだけで判断できる**情報密度
  - 通知は alert のみ即時、warn/info は日次ダイジェスト
  - 初期設定ウィザードで 10 分で開始できる
- **検知**: 教員の利用時間、ダッシュボード滞在時間
- **回復**: 運用ドキュメント、先生向けオンボーディング動画(Phase 4)

### R-14: 家庭環境の格差(タブレット不在、通信環境)
- **Impact**: M — 教育機会の不均衡
- **Likelihood**: H
- **予防**:
  - PC でも動く UI(ただし最適は iPad / Chromebook)
  - 音声が前提ではない(テキスト入力でも完結)
  - PWA でオフライン最低限動作
- **検知**: 家庭からのアクセス率
- **回復**: 学校内での活動を完結させる(宿題化しない)

### R-15: AI 依存で子ども自身の思考力が育たない
- **Impact**: M — 教育目的の自己否定
- **Likelihood**: M
- **予防**:
  - 「AI に**教える**」体験をコアに据える(受け身にさせない)
  - ナレッジ外質問への「一緒に調べよう」誘導
  - 連続使用警告で休憩を促す
- **検知**: 1 児童あたりの「ナレッジ追加回数 / 対話回数」比(高いほど探究)
- **回復**: 比率が低い児童には「調べるチャレンジ」を表示(Phase 4)

### R-16: ログ・監査データから児童の個人情報が漏れる
- **Impact**: M
- **Likelihood**: L
- **予防**:
  - `ModerationLog.pendingInputEncrypted` は暗号化
  - Sentry / PostHog への送信前にフィルタ
  - ログ保持期間の設定(`AUDIT_LOG_RETENTION_DAYS=365`)
- **検知**: ログのサンプル監査(四半期ごと)
- **回復**: 該当ログを削除、インシデント記録

## 🟢 低優先リスク(把握しておく)

### R-17: 児童が AI キャラクターに感情的に依存
- **Impact**: L〜M
- **Likelihood**: L
- **対策**: ペルソナを「学習ボット」として設計(友だちではない)、連続使用警告

### R-18: ブランドの商標侵害(アバター・UI 絵柄)
- **Impact**: L
- **Likelihood**: L
- **対策**: 独自イラスト、商業キャラクターの使用禁止をデザインガイドに明記

### R-19: 画像 API の規約変更で機能停止
- **Impact**: M
- **Likelihood**: L
- **対策**: Adapter でフォールバックチェーン、代替プロバイダを常に 1 つ用意

### R-20: SQLite の破損・同時書き込み競合
- **Impact**: M
- **Likelihood**: L(教室規模 30 名では稀)
- **対策**: WAL モード、日次バックアップ、Postgres 移行の選択肢

## 🔬 研究機能に関するリスク(Phase 2 以降で随時対処)

### R-21: AI の偏りの教材化が「AI を信じてはいけない」の単一メッセージに矮小化
- **カテゴリ**: 教育的
- **Impact**: H — 本アプリの核が失われる
- **Likelihood**: M
- **予防**:
  - [missing-voice-probe.md](04-prompts/missing-voice-probe.md) の禁則で「AI を信じるな」と直接言わない
  - invitation は必ず**問いの形**で、児童に判断を戻す
  - 画面 19 の UI で、AI 候補を「仮説」と明示
- **検知**:
  - 児童の `MissingVoiceHypothesis` の記述が単調化していないか、教員が画面 15 でサンプル読解
- **回復**:
  - 事後の教員研修で教材化の意図を再共有

### R-22: 立ち止まりの言葉の「量的評価化」が児童を縛る
- **カテゴリ**: 教育的
- **Impact**: H — 児童が「立ち止まり語」を**狙って**使い始めると、研究指標としても学習としても機能不全
- **Likelihood**: M
- **予防**:
  - 画面 20 のセルフビューで**成績・評価に使わない**明示
  - 教員ダッシュボードに「量だけで評価しない」警告を常設
  - Wilcoxon 結果には必ず cautions を表示
- **検知**:
  - 児童の文章長あたりの標準化率が急上昇したら警告
- **回復**:
  - 対話や振り返りの**質的読み**を教員研修で重視

### R-23: 研究データの収集範囲の誤解・同意の不徹底
- **カテゴリ**: 倫理・法令
- **Impact**: C — 研究倫理違反
- **Likelihood**: L(設計で抑えるが、運用依存)
- **予防**:
  - `ConsentRecord(kind='research-participation')` を独立軸に
  - `Unit.researchMode=true` は同意が揃うまで ON にできない構造制約
  - 画面 13 の公開前チェックリストに「倫理承認を確認した」チェック必須
- **検知**:
  - `Unit.ethicsApproval` が空のまま researchMode が ON のケースを CI/監査で検出
- **回復**:
  - 発覚時は即 researchMode を OFF、該当期間のデータを物理削除、インシデント報告

### R-24: エピソード記述の PII 復活(教員編集段階)
- **カテゴリ**: プライバシー
- **Impact**: H
- **Likelihood**: L(教員の不注意での混入)
- **予防**:
  - 画面 15 のエディタに**保存時の PII スキャン**
  - 教員への説明で「原本(復号済み)を編集せず、要約を書き直す」運用を徹底
- **検知**:
  - `EpisodeRecord.narrative` の保存時に pii ルールベース検出 + LLM 検査
- **回復**:
  - 検出時は保存拒否、教員に説明

### R-25: アンケート設問の潜在的誘導
- **カテゴリ**: 研究妥当性
- **Impact**: H — 測定そのものが歪む
- **Likelihood**: M — 善意の教員による意図せぬ誘導も
- **予防**:
  - [pre-post-survey-gen.md](04-prompts/pre-post-survey-gen.md) の中立性制約
  - 画面 13 で `neutralityCheckNotes` を読むチェック必須
  - 承認前に**指導教員/同僚**の目を通す運用(アプリで共有リンク付与)
- **検知**:
  - 代表的な誘導語の正規表現検査(「べき」「正しい」「間違った」等)
- **回復**:
  - 公開後でも設問の差替え可、その場合は pre/post の比較性が失われる警告を表示

### R-26: 「AI に出てこないのは誰?」が結論提示に逆転
- **カテゴリ**: 教育的
- **Impact**: H
- **Likelihood**: M
- **予防**:
  - プロンプトで**仮説として提示**することを強制
  - UI では「AI がそう言ったから」ではなく「あなたはどう思う?」を繰り返す
- **検知**:
  - `MissingVoiceHypothesis.hypothesisText` が AI 応答のコピペになっていないか文字一致で検出
- **回復**:
  - 一致率が高い児童には「あなたの言葉で書いてみよう」ガイドを表示

### R-27: 対話ログ返却時のプライバシー配慮(保護者経由)
- **カテゴリ**: プライバシー
- **Impact**: M
- **Likelihood**: M
- **予防**:
  - 返却 PDF は児童 1 人分のみ、他児童の発話を含めない
  - 保護者への引き渡しは教員の手を介する運用(アプリ内で保護者に直送しない)
- **検知**:
  - 返却 PDF 生成時に、対象児童以外の `userId` が混入しないかテスト
- **回復**:
  - 誤配の可能性があれば教員に通知、当該ファイルの配布停止

## 📋 リスク管理のプロセス

### レビュー頻度
- **リリース前**: 全件レビュー、対策の実装確認
- **月次**: `IncidentReport` を集計、新しいリスク追加
- **四半期**: リスク一覧を再評価、優先度の見直し

### インシデント学習
- `IncidentReport (severity: alert)` が起きたら、類似を未然に防ぐ仕組みを**コードに**落とす
- ポストモーテムは教員と開発の合同で

### 責任分担
| 層 | 主責任 |
|----|-------|
| プロンプト設計 | 開発チーム |
| モデレーション運用 | 開発チーム + 教員 |
| 児童への対応 | 教員 |
| 学校全体・保護者対応 | 学校管理職 |
| 法令・規約 | 開発チームのリーガル相談先 |
